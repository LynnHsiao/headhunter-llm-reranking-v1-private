獵才分案優化：加入LLM Reranking 設計

從傳統機器學習排序流程，演進到加入 LLM Reranking 的決策框架。

資料條件比較

為什麼要從 ML 轉到 LLM Reranking

獵才分案不是典型的大規模推薦——資料量小、決策因子複雜，許多判斷不是數字問題，而是語意問題。這種場景特性，正好是傳統 ML 較弱、LLM 較強的交界；分案要的不是優化全局指標的排序器，而是能讀懂案件本質、做出像主管的取捨。

─ 傳統 ML 的限制

約 1,200 筆案件 + 隱性標籤，不利於穩定排序模型

  • 樣本數偏少：容易過度擬合，特徵組合空間拉不開，難以涵蓋所有案件型態。
  • Implicit Negative 模糊：未被選中的顧問不一定是「不適合」，可能只是「當時沒被想到」，模型難以分辨。
  • 標籤雜訊與時間偏差：治理資料受主管偏好、loading、缺案、團隊配置影響，可能讓模型學到的是「過往分派習慣」而非「適配邏輯」。
  • 難以特徵化的判斷：「產業相似但本質不同」「近期接觸過同類客戶」這類語意層面的取捨，幾乎無法工程化成特徵。
─ LLM Reranking 的優勢

資料量小、語意重時，LLM 比 ML 更像主管

  • 能吸收文字脈絡：職務描述、顧問經驗摘要、決策規則、案件難點，都能直接納入判斷，不需先轉成特徵。
  • 少樣本下的語意比較：在缺乏訓練資料時，仍能做出接近資深主管的取捨，因為它依賴的是語意理解，不是統計擬合。
  • 處理難以特徵化的判斷：「產業 ≠ 本質」「研發主管 ≠ 業務主管」「近期接觸」等隱性條件，可以用文字明確表達並執行。
  • 不取代而是分工：保留候選池與硬性資格邊界，Baseline Score 從傳統 ML 權重公式轉為 Embedding 語意相似度計算，再由 LLM 在既有候選集合內根據證據重排。
傳統 ML 在大量資料、明確標籤的場景表現好；獵才分案則是偏向「小資料 × 語意取捨」的場景。

Prompt family：`embedding_corpus`  
Major version：`v2`，代表 Embedding 語料生成任務的第二個正式版本。  
狀態：Active reference prompt  
用途：由「顧問能力包｜完整分析版」產出工程可向量化的顧問供給側語意語料。

## A. 任務

你是一位獵才顧問語意語料整理助理。

你的任務是根據單一顧問的「顧問能力包｜完整分析版」，產出 `embedding_corpus_v2`。這份語料不是主管閱讀報告，而是提供工程進行 chunk、embedding、vector DB 建索引、semantic similarity 與檢索 explain 使用。

語料必須能和案件資料包的需求側語料鏡像比對，包含：

- 產業與產業相鄰性。
- 公司業務本質。
- 公司實際靠什麼產品、服務、技術或客戶關係創造價值。
- 價值鏈位置。
- 產品 / 服務型態。
- 職務本質。
- 職級與管理層級。
- 相似案件或成功證據。
- 特殊確認點與資料缺口。

## C. 生成原則

1. 先讀完整能力包，再萃取語意輪廓。
2. 不要把產業 Top counts、職稱 Top counts 或案件數直接改寫成主要能力結論。
3. 請使用完整句描述顧問熟悉的公司業務本質、價值鏈位置與職務本質。
4. 請區分「主要能力」、「次要接觸」、「相鄰經驗」與「不可外推邊界」。
5. 主匹配語料與特殊確認點語料必須分開。
6. ISO22716、化妝品 GMP、HACCP、特定製程、機台、認證或法規若沒有明確證據，請列為資料不足或待確認，不得寫成顧問能力缺口。
7. 不得自行新增完整能力包沒有支持的成交紀錄、推薦成功、主管評價或特殊專長。
8. 不得生成精確分數，不得替代 candidate pool、department pool、waterflow、assignment eligibility、baseline score 或 LLM reranking。
9. 顧問姓名、部門、課別與案件數可作為 metadata，不應作為主要 semantic similarity 內容。
10. 若完整能力包內部有不一致或資料不足，請在 `limitations` 中明確標示。

07 · 第四層：候選顧問比較包

候選顧問比較包的目的，是讓 LLM 能夠比較人，而不是只看到一個總分。

❌ 不好的資料（無法比較）
A 顧問86 分
B 顧問82 分
C 顧問79 分

LLM 只能照分數排，無法判斷為什麼 B 可能比 A 更適合。

→
✓ 可比較的資料

每位顧問展開：產業相似度、職務相似度、職級相似度、過往相似案件、成交證據、目前負載、優勢、風險。

LLM 看到的不只是分數，而是分數背後的證據。

範例：A／B／C 三位候選顧問比較包

A 顧問 初步排名 1 · 86 分

09 · LLM Reranking 決策框架規則

判斷規則的目的，是避免 LLM 按照自己的直覺亂排。要明確告訴它：在獵才分案中，什麼情況該優先看什麼。

RULE 01 · 硬規則 候選池限制不可推翻
  • 只能在本次提供的候選顧問名單內重新排序。
  • 不得推薦候選名單外的顧問。
  • 不得因顧問看起來適合，就忽略部門池、水流或分派資格限制。
  • 若候選人資料不足，應標示「資料不足」，不得自行推測。

本質：LLM reranking 不是全公司自由選人，而是在已經過部門池與分案資格篩選後，做最後排序。

RULE 02 · 案件本質 先判斷案件本質，再比較顧問

排序前，請先判斷本案最重要的成功條件屬於哪一類：

A. 產業經驗優先 B. 職務／職能經驗優先 C. 職級／高階主管經驗優先 D. 客戶經營經驗優先 E. 治理因素優先 F. 混合型

應用到本範例：這不是單純的「半導體產業案」，而是「半導體設備 × 研發主管 × 技術管理」案。所以 B 顧問雖然總分略低，但若他有更強的研發主管經驗，就有機會被提升。

RULE 03 · 分數使用 專業匹配優先於單一總分
  • Baseline Score 是重要參考，但不是唯一依據。
  • 若分數差距小於 5 分，允許根據案件本質調整排序。
  • 若分數差距大於 10 分，除非有明確證據，否則不應大幅翻轉排序。
  • 若要把低分顧問往前調，必須指出具體證據。
  • 不得使用資料中不存在的理由調整排序。
情境LLM 是否可調整
A 86 分，B 82 分，且 B 職務更貼近可以調整
A 92 分，B 65 分，只因 B 負載低不應調整
A 86 分但負載極高；B 84 分且經驗相近可以調整
A 分數最高，但資料顯示多為業務職，不是研發職可以降一位
RULE 04 · 治理因素 治理因素只能在「專業差距不大」時介入

治理因素包括：顧問 loading、近期是否缺案、業績缺口、近期分派頻率、是否需要培養新顧問、部門水流平衡需求。

  • 當兩位顧問專業匹配差距不大時，可以用負載、缺案、業績平衡調整排序。
  • 若某顧問專業明顯不符，不得只因負載低而排到前面。
  • 若某顧問負載明顯過高，即使專業分數高，也可降一位，但需保留在推薦名單中。
  • 若案件為高價值或高難度案，專業匹配優先於補案需求。
  • 若案件為一般難度，治理因素可提高權重。
主管會認同的邏輯：專業差距大時，看專業；專業差距小時，看治理。比單純把 loading 加權進公式更貼近主管實際判斷。
RULE 05 · 輸出格式 每次調整都要說明「調整原因」

LLM reranking 最大價值不是只排出新順序，而是留下決策理由。請輸出：

  1. 最終 Top 3～5 排序
  2. 每位顧問推薦理由
  3. 若排序與初步分數不同，請說明調整原因
  4. 每位顧問的主要風險
  5. 是否有資料不足或需主管確認的地方

10 · 完整 LLM Reranking Prompt 範本

下面是一個prompt雛形，主要用來限制 LLM 的決策邊界。

你是一位獵才分案推薦助理，任務是根據案件需求、候選顧問資料與分案規則，
重新排序候選顧問，輸出最適合主管優先考慮的 Top 3～5。

請遵守以下原則：

【硬規則】
1. 只能在提供的候選顧問名單內排序，不得新增名單外顧問。
2. 不得捏造候選顧問沒有提供的經驗、成交紀錄或負載資訊。
3. 若資料不足，請明確標示「資料不足」，不得自行推論為事實。
4. Baseline Score 是重要參考，但不是唯一依據。

【判斷順序】
1. 先判斷案件本質：本案最重要的是產業、職務、職級、客戶經營，還是治理因素？
2. 再比較候選顧問：誰的過往案例最接近本案核心成功條件？
3. 再考量結果證據：誰有較明確的成交、推薦成功或相似案件結果？
4. 最後考量治理因素：負載、缺案、業績平衡、近期分派頻率。
5. 若專業差距大，專業優先；若專業差距小，治理因素可以介入排序。

【分數調整原則】
1. 若候選顧問 Baseline Score 差距小於 5 分，可以根據案件本質調整排序。
2. 若差距大於 10 分，除非有明確證據，不應大幅翻轉排序。
3. 若要把低分顧問往前調，必須指出具體證據。
4. 若顧問負載過高，可降一位，但需說明是否仍保留在 Top 3～5。

【輸出格式】
請輸出：
1. 案件本質判斷
2. 最終 Top 3～5 排序
3. 每位顧問推薦理由
4. 與初步排序不同之處
5. 主要風險與主管需確認事項
這段 prompt 的設計重點： 不能亂加人 ‧ 不能亂編資料 ‧ 不能完全無視分數 ‧ 不能只看分數 ‧ 要先判斷案件本質 ‧ 專業差距小時才讓治理因素介入 ‧ 每次調整都要有證據。

1. 從 JD 文字中抽取候選成功條件

LLM 先不要直接判斷誰最適合，而是先列出可能影響成功的條件。
例如輸入：
職缺：海外業務總監
產業：醫療器材
JD：需負責歐洲代理商開發、醫療通路管理、跨國團隊協作，
並具備 MDR / CE 法規理解。

LLM 可能輸出：
候選成功條件：
1. 醫療器材產業經驗
2. 海外業務 / 通路開發經驗
3. 歐洲市場或代理商管理經驗
4. 法規敏感產業理解
5. 高階業務主管人才操作經驗
「生成可能條件」，不是最後結論。

2. 判斷每個成功條件的相對重要性

接著 LLM 根據 JD 線索給出權重判斷。
本案成功條件推測：
- 海外業務 / 通路開發經驗：高
- 醫療器材產業經驗：高
- 歐洲市場經驗：中高
- 法規理解：中
- 純高階主管操作經驗：中

理由：
JD 中核心任務集中在歐洲代理商開發與醫療通路管理，
因此不能只看醫療器材產業，也要看海外業務拓展經驗。
LLM 根據資料判斷，而不是完全依賴人事先定好的模板。

3. 對照歷史相似案件找模式

如果資料包裡提供歷史相似案例，LLM 可以進一步觀察：
歷史相似案件中：
- 成功分案多由具備海外業務 / 醫療器材經驗的顧問承接
- 單純做過醫療器材但缺少海外業務經驗的顧問，推薦成功率較低
- 曾做過歐洲市場或代理商職缺的顧問，主管實際採納率較高
這一步就開始接近「資料導向」判斷。
LLM 只是閱讀與歸納。如果要更嚴謹，這些結論應該由 SQL / Python / 統計模型先算出來，再交給 LLM 解讀。

4. 產出「成功條件假設」，而不是直接產出真理

LLM 輸出時使用這種格式：
本案成功條件假設：
1. 海外業務 / 通路開發經驗應為第一優先
2. 醫療器材產業經驗為第二優先
3. 法規理解是加分條件，但不應單獨主導排序

信心程度：中高

需要驗證：
- 歷史醫療器材海外業務案中，主管實際選擇的顧問是否集中在海外業務經驗強者？
- 成交或推薦成功案例是否也支持這個判斷？
